UML was developed by Grady Booch, Ivar Jacobson and James Rumbaugh (The Three Amigos) during 1994-1995. Latest version 2.5.1 released in December 2017 by Object Management Group (OMG). UML is a standard object oriented documentation models for any software system. 2.5 version written to increase the clarity and to remove the redundancy.

  1. Things

    • Structural : Class, Object, Interface, Collaboration, Use case, Actor, Component, Node, Deployment
    • Behavioral : State machine, Activity diagram, Interaction diagrams
    • Grouping : Package
    • Annotations : Annotation

  2. Relationships

    • Association : Connects 2 object and defines how many objects are associated with.
    • Dependency : Relation depends on each objects.
    • Generalization : Inheritance between objects. sub class and parent class relation.
    • Realization : Relation with interface to implement it's function.
    • Reflection : Subtype of association. represents the relation of instances of same class.
    • Direction : Association from one to another flows for one direction only.
    • Composition : Not a UML association.Binary whole part association.Sub type of aggregation.
    • Aggregation : Sub type of association.Whole part, even part remain after whole deleted.

  3. Diagrams

    4. Three types of diagrams will explain in next section.

  1. Structure Diagrams

    Represent the static view of the system.Describes the objects in the system.

    • Class
    • Component
    • Object
    • Composite
    • Deployment
    • Package
    • Profile

  2. Behavior Diagrams

    3. Represent the dynamic functioning of the system.

    • Activity
    • Use case
    • State Machine

  3. Interaction Diagrams

    4. Subset of behavioral diagrams. Represents the data flow/interaction between use cases/entities.

    • Sequence
    • Communication
    • Interaction
    • Timing

  1. Class diagram

    2. Class must have class name, attributes, and operations. Class name should be start with capital letter, bold( italic for abstract ) , and centered text in first rectangle. Attributes adding after class name. Public, private, protected and package visibility has +, -, #, ~ sings. Derived attributes are which can computed from another. ex. age from date of birth.

    Class diagram use in different stages of software development life cycle for conceptual, specification and development perspectives.

  2. Object diagram

    3. Class diagram represents abstract view, but object diagram describes the instance of a class and use to analyse the behaviour of a class.

    Use for forward/reverse engineering of systems, analyse the offline/online systems.

  3. Component diagram

    4. Model the subsystems of a system.

  4. Deployment diagram

    5. Model the software deployment design on hardware system.

  1. Use case diagram

    2. Use to capture the functionality, dynamic live behaviour and the requirements of a system. Consist with use cases, actors who responsible for invoke the functions, external thing/person interact with the system and annotations. Use case represent a distinct functionality.

  2. State machine/chart/transition diagram

    3. Model the state of each object, package, and entire system and how system interact with the external world. Two types : behavioral and protocol state machines.

    When state carts model a state of live system with WAIT concept, Flow carts model the data flow of a system with controls without using the WAIT concept.

  3. Activity diagram

    4. Describes the execution flow of the system Or the flow cart of the activities between entities. Use activity, activity partitions, fork and join nodes, pins to drow the diagram.

Describes how the message passing and life-line between entities to form up the activities of the system. Life line has a name (optional), type, selector (optional). Messages are divided as synchronous, asynchronous, return, object creation, object destruction, found, lost messages. State invariants and constrains use to show the change of a object state after a message, event, operation within object instance's life time for this diagrams. Operators use show how operands are executes. Operations types are Opt (option), Alt (alternative), Loop, Break, Par (parallel), Ref (reference).

  1. Sequence diagram

    2. Shows interaction of messages between life-lines.

  2. Collaboration diagram

    3. Shows structural aspects with life-lines.

  3. Timing diagram

    4. Shows message interaction between instances.
